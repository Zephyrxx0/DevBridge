# Architecture

**Analysis Date:** 2024-05-24

## Pattern Overview

**Overall:** AI-Augmented Multi-Tier Web Application (RAG pattern)

**Key Characteristics:**
- **Decoupled Frontend/Backend:** Next.js application interacting with a Python FastAPI backend.
- **RAG (Retrieval-Augmented Generation):** The system indexes repositories and uses a vector database to augment AI interactions with semantic search.
- **Agentic Orchestration:** Employs LangChain and LangGraph for reasoning, code search, and synthesizing history/annotations.
- **Event-Driven & Streaming:** Utilizes Server-Sent Events (SSE) for real-time generative responses and Pub/Sub (indicated by infra) for background ingestion tasks.

## Layers

**Frontend Layer:**
- Purpose: Provides user interface for repository management, chatting, visual codebase exploration (Knowledge Map).
- Location: `web/`
- Contains: Next.js App Router setup, React components (Radix UI/shadcn), Tailwind CSS.
- Depends on: Backend APIs, Next.js routing, React context.

**API Layer:**
- Purpose: Serves incoming requests from frontend, orchestrates data logic.
- Location: `api/`
- Contains: FastAPI application (`api/main.py`), routing logic, server-sent events for streaming, and caching decorators.
- Depends on: PostgreSQL (Supabase), LangChain/LangGraph, Vertex AI.

**Agent & Orchestration Layer:**
- Purpose: Employs AI to interpret questions, fetch codebase context via vector database, and generate grounded answers.
- Location: `api/agents/`
- Contains: LangGraph state machines, prompt logic, and tool bindings (e.g., `code_search`, `search_pr_history`, `get_pr_detail`).
- Depends on: Vertex AI (Gemini), `api/db/vector_store.py`.

**Ingestion & Data Pipeline Layer:**
- Purpose: Parses codebases, chunks files via Tree-sitter, creates embeddings, and updates the vector database.
- Location: `api/ingestion/` & `api/ingest/`
- Contains: Source discovery, AST parsing, chunk generation.

**Infrastructure Layer:**
- Purpose: Provisions cloud resources.
- Location: `infra/`
- Contains: Terraform configurations for Cloud Run, Google Cloud Storage, and Pub/Sub.

## Data Flow

**Chat & Code Query Flow:**
1. User submits a query in the frontend UI (`web/`).
2. Next.js passes the request to the FastAPI endpoint (`POST /chat/stream`).
3. The `Orchestrator` (`api/agents/orchestrator.py`) processes the query in LangGraph.
4. The AI uses tools (e.g., `code_search`) to query the vector database (`api/db/vector_store.py`) via pgvector.
5. Code chunks, PR history, and annotations are injected into the context.
6. A response with citations is generated by Google Vertex AI and streamed back to the client via SSE.

**Repository Ingestion Flow:**
1. Code repository is uploaded or targeted.
2. The `api/ingestion/pipeline.py` discovers files and extracts structured chunks using `api/ingestion/tree_sitter_chunker.py`.
3. Embeddings are generated.
4. Chunks and embeddings are stored in Supabase with pgvector.

## Key Abstractions

**Orchestrator (`api/agents/orchestrator.py`):**
- Purpose: Wraps LangChain models and tools into an agentic LangGraph execution environment. Uses tools like `code_search` to augment generation.
- Pattern: ReAct (Reasoning and Acting) Agent.

**Vector Database Manager (`api/db/vector_store.py`):**
- Purpose: Abstracts underlying pgvector querying. Uses hybrid search (semantic + metadata).
- Pattern: Repository/Data Access Object.

**Chunking Pipeline (`api/ingestion/pipeline.py` & `tree_sitter_chunker.py`):**
- Purpose: Converts arbitrary codebase directories into standardized `CodeChunk` objects for semantic indexing.

## Entry Points

**API Service:**
- Location: `api/main.py`
- Triggers: Uvicorn HTTP server execution.
- Responsibilities: Routing, CORS, injecting auth contexts, and mounting routes/cache.

**Web Application:**
- Location: `web/src/app/page.tsx` & `web/src/app/layout.tsx`
- Triggers: Next.js edge/Node server responding to user HTTP requests.
- Responsibilities: Server-side rendering, client initialization.

## Error Handling

**Strategy:** Exception propagation with standardized HTTP responses in the API, and graceful fallbacks in AI generation (e.g., MockLLM when Vertex AI isn't configured).

**Patterns:**
- Try/Catch blocks around async tool calls with `asyncio.wait_for` timeouts to prevent LLM stalls.
- FastAPI `HTTPException` for routing errors.
- Streaming endpoints yield localized `{"type": "error"}` SSE messages rather than dropping connections.

## Cross-Cutting Concerns

**Authentication:** Handled via middleware (`api/main.py`) intercepting `X-Internal-Auth` tokens against pre-configured trust IPs.
**Caching:** Uses `fastapi-cache` (`api/db/cache.py`) backed by PostgreSQL for deduplicating API responses.
**Database Connection:** Asynchronous SQLAlchemy engines via connection pooling (`api/db/session.py`), normalizing connection strings on startup.
