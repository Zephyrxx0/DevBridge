# Phase 25: Task Scheduling - Context

**Gathered:** 2026-05-16
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase implements a robust task scheduling system within the FastAPI backend. It leverages `APScheduler` to orchestrate recurring background jobs such as data synchronization, cache maintenance, metrics collection, and automated report generation. It focuses on persistence, reliability (audit trails/retries), and concurrency safety (distributed locking).

</domain>

<decisions>
## Implementation Decisions

### Scheduling Architecture
- **D-01: Persistent DB-backed Job Store.** Jobs will be stored using `SQLAlchemyJobStore` in the existing Postgres/Supabase database. This ensures that job definitions survive application restarts and can be managed dynamically.
- **D-02: Internal APScheduler Integration.** The scheduler will run internally within the FastAPI application lifecycle (using `lifespan` events). This matches the requirement for a simpler stack without external dependencies like Redis/RQ.
- **D-03: Distributed Locking (DB-based).** To prevent duplicate job execution in a multi-instance (Cloud Run) environment, a database-level locking mechanism (e.g., Postgres advisory locks) will be implemented to ensure only one instance executes a given job at a time.

### Job Management & Visibility
- **D-04: Audit Table + Retry Logic.** A dedicated `job_history` table will be created to track job executions, including status, duration, and error logs. A configurable retry policy will be implemented for failed sync and report jobs.
- **D-05: Reports Hub.** A new "Reports Hub" module will be established to store and serve generated reports (Daily/Weekly).

### Job Inventory
- **D-06: Core Jobs List.**
  - **Daily Sync**: Fetching updates from GitHub and external docs.
  - **Cache Cleanup**: Pruning the local repository cache and GCS temporary files.
  - **Metrics Collection**: Aggregating system usage and performance data.
  - **Report Generation**: Automated daily and weekly intelligence reports for the Reports Hub.

### Claude's Discretion
- The specific table schema for `job_history`.
- The exact implementation of the Postgres advisory lock wrapper.
- The directory structure for report storage (e.g., `/app/reports` bind-mounted or GCS-backed).

</decisions>
