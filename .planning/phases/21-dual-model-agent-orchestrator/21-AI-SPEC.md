## 1. System Classification

**System Type:** Multi-Agent

**Description:**
A dual-model agentic system that partitions reasoning labor across two different local models running on a single AMD MI300X GPU. A Fast Model (Gemma-4) acts as an intent classifier and supervisor, routing queries to either itself (for trivial/fast tasks) or a Big Model (Qwen2.5-72B) for deep codebase analysis and complex reasoning. The system prioritizes low latency for simple queries while maintaining high reasoning quality for deep technical questions, all while staying within a strict 192GB VRAM budget.

**Critical Failure Modes:**
1. **Routing Oscillation/Incorrectness:** Fast Model incorrectly classifies a complex query as FAST, leading to shallow/hallucinated technical answers.
2. **Resource Starvation (VRAM OOM):** Failure to strictly enforce memory partitions or context caps, causing one model to crash the other on the shared GPU.
3. **Fallback Failure:** System hangs or returns opaque errors when the Big Model times out (120s), instead of gracefully failing over to the Fast Model with a UI warning.

## 1b. Domain Context

**Industry Vertical:** Software Engineering / Developer Productivity (DevEx)
**User Population:** Software Engineers, DevOps Engineers, System Architects
**Stakes Level:** Medium to High
**Output Consequence:** AI suggestions directly influence codebase quality, security posture, and developer velocity. Incorrect outputs lead to technical debt, security vulnerabilities (CWE), or developer frustration (cognitive load).

### What Domain Experts Evaluate Against

Dimension: Contextual Grounding
Good (domain expert would accept): Response utilizes existing internal abstractions, naming conventions, and architectural patterns (e.g., uses the project's `CustomLogger` and `BaseRepository` instead of generic `print` or raw SQL).
Bad (domain expert would flag): Response suggests a generic solution that ignores the current codebase structure or introduces redundant libraries (Context Blindness).
Stakes: High
Source: Practitioner knowledge — maintaining architectural consistency is a primary duty of senior engineers.

Dimension: Routing Precision (Cost/Latency vs. Quality)
Good: Simple greetings or UI clarifications are handled by the Fast model (<200ms); complex cross-file refactoring or performance optimization queries are routed to the Big model.
Bad: A request to "Optimize this complex recursive function" is handled by the Fast model, resulting in a shallow, incorrect, or hallucinated suggestion.
Stakes: Medium
Source: Research (Google/Anthropic) — balancing "Flow State" (low latency) with "Deep Reasoning" (high accuracy).

Dimension: Security & Robustness (CWE Adherence)
Good: Code suggestions include input validation, avoid common pitfalls (e.g., SQL injection, insecure deserialization), and follow the principle of least privilege.
Bad: AI suggests using `eval()` on user input or ignores existing authentication middleware when adding a new endpoint.
Stakes: Critical
Source: Security Standards (OWASP/CWE) — AI-generated code must pass the same security gates as human code.

Dimension: Developer Friction (Cognitive Load)
Good: The system provides clear feedback during long-running "Deep" reasoning tasks (e.g., "Analyzing repository structure...") and clearly flags when it has fallen back to "Fast Mode" due to a timeout.
Bad: The UI hangs for 120 seconds with no progress indicator, or the system silently returns a low-quality "Fast" response after a "Deep" failure without notification.
Stakes: Medium
Source: Developer Experience (DX) research — uncertainty and lack of transparency break the "Flow State."

### Known Failure Modes in This Domain

- **Context Window Exhaustion:** Attempting to analyze a 100-file PR by stuffing every file into the Big Model's prompt, exceeding the 48K token cap and causing truncation that deletes the most relevant files.
- **Stale Context Hallucination:** The agent references a class or function that was deleted or renamed in a recent commit because its internal knowledge graph or index is out of sync.
- **Dependency Drift:** Suggesting code that requires a library version incompatible with the project's current `package.json` or `requirements.txt`.

### Regulatory / Compliance Context

- **Data Sovereignty:** All inference must remain on-premises (AMD MI300X) to comply with corporate security policies regarding IP protection.
- **CWE/OWASP Compliance:** AI-generated code must not introduce "Known Vulnerabilities" (CWE Top 25).
- **License Compliance:** Generated code must not violate GPL/MIT/Apache licenses (though local models mitigate risk, "hallucinated licenses" in comments can cause legal friction).

## 2. Framework Decision

**Selected Framework:** LangGraph

**Version:** Pin to latest stable (v0.2.x / v1.1.x ecosystem)

**Rationale:**
LangGraph is the industry standard for controlled, stateful multi-agent orchestration. It natively supports the Supervisor pattern required for this dual-model setup. Crucially, its node-level checkpointing and explicit state management satisfy the project's persistence requirements and allow for reliable fallback logic between nodes (models). Since the codebase already uses LangChain, LangGraph offers the lowest integration friction while providing the necessary guardrails for hardware-constrained inference.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| PydanticAI | While offering superior type-safety, it lacks the native graph-based persistence and fine-grained node-level control for hardware-specific fallbacks that LangGraph provides out of the box for this specific legacy-integrated stack. |

**Vendor Lock-In Accepted:** No — The system is model-agnostic and runs on local AMD hardware (ROCm/vLLM).

### Key Imports
```python
from typing import Annotated, TypedDict, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver # For local dev
from langchain_openai import ChatOpenAI
from langchain_core.messages import BaseMessage, HumanMessage
```

### Minimal Entry Point (Supervisor/Router Pattern)
```python
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], operator.add]
    next: str

def router(state: AgentState):
    # D-01: Returns "FAST" or "DEEP"
    decision = fast_model.invoke(state["messages"]) 
    return {"next": "big_worker" if "DEEP" in decision.content else "fast_worker"}

builder = StateGraph(AgentState)
builder.add_node("router", router)
builder.add_node("fast_worker", fast_worker_node)
builder.add_node("big_worker", big_worker_node)

builder.add_edge(START, "router")
builder.add_conditional_edges("router", lambda x: x["next"])
builder.add_edge("fast_worker", END)
builder.add_edge("big_worker", END)

graph = builder.compile(checkpointer=MemorySaver())
```

### Abstractions
| Concept | Description |
|---------|-------------|
| **State** | Shared data structure updated by nodes via reducers (e.g., `operator.add`). |
| **Node** | Python function/runnable performing work (Fast vs Big model calls). |
| **Edge** | Control flow paths; **Conditional Edges** enable the routing logic. |
| **Checkpointer** | Persists state for multi-turn chat and crash recovery. |
| **Command** | Object returned by nodes to explicitly control the next step (v0.2+). |

### Pitfalls
- **State Mutation:** Never mutate `state` in-place. Always return a dict of updates.
- **Async Mismatch:** Mixing `invoke()` and `ainvoke()` in the same graph can lead to event loop deadlocks in complex nested graphs. Use `astream`/`ainvoke` throughout.
- **Infinite Loops:** Without a `max_iterations` or a clear terminal `END` path, graphs can loop indefinitely on local hardware, burning VRAM.
- **Large State:** Avoid storing large raw strings (like whole files) in `state["messages"]`. Use specialized keys for large context to avoid bloating checkpoints.

### Folder Structure
```text

├── state.py        # AgentState definition
├── graph.py        # Graph compilation & Router logic
├── nodes/
│   ├── fast.py     # Fast Model worker node
│   └── big.py      # Big Model worker node
└── utils/
    └── fallback.py  # Error handling & Fallback wrappers
```

## 4. Implementation Guidance

### Model Configuration
- **Fast Model (Gemma-4):** `base_url="http://localhost:8001/v1"`, `max_tokens=256`, `temperature=0.0`. Optimized for routing latency.
- **Big Model (Qwen2.5-72B):** `base_url="http://localhost:8000/v1"`, `max_tokens=4096`, `temperature=0.7`. Configured with tool-calling capabilities for repo analysis.

### Core Routing Pattern
The `router` node must handle the **D-01 Binary Toggle**.
```python
async def intent_classifier(state: AgentState):
    prompt = "Classify the user intent. Output 'FAST' for greetings/clarifications or 'DEEP' for code analysis/complex logic. [Query]: " + state["messages"][-1].content
    response = await fast_llm.ainvoke(prompt)
    choice = "big_worker" if "DEEP" in response.content.upper() else "fast_worker"
    return {"next": choice}
```

### Hardware-Specific Fallback (D-03/D-04)
Wrap the Big Model call in an `asyncio.wait_for` to enforce the 120s timeout. On `TimeoutError`, return a state update with `fallback=True` and route to the `fast_worker` node.

```python
async def big_worker_node(state: AgentState):
    try:
        # Enforce IR-01/D-04 timeout
        result = await asyncio.wait_for(big_llm.ainvoke(state["messages"]), timeout=120)
        return {"messages": [result]}
    except (asyncio.TimeoutError, Exception) as e:
        # D-03: Signal UI for fallback badge
        return Command(
            update={"messages": [HumanMessage(content="[SYSTEM: FALLBACK TRIGGERED] " + str(e))], "fallback": True},
            goto="fast_worker"
        )
```

## 4b. AI Systems Best Practices

### 4b.1 Structured Outputs with Pydantic
Even though D-01 uses simple text for routing, structured output is mandatory for extraction tasks (Phase 08+).
- **Example:**
```python
class AnalysisResult(BaseModel):
    summary: str = Field(description="Brief summary of findings")
    vulnerabilities: list[str] = Field(default_factory=list)
    complexity_score: int = Field(ge=1, le=10)

structured_llm = big_llm.with_structured_output(AnalysisResult)
```
- **Framework Integration:** LangGraph handles this by returning the Pydantic object as the node's update value.
- **Retry Logic:** Use `langchain_core.runnables.RunnableRetry` to attempt 2-3 retries on parsing errors before surfacing to UI.

### 4b.2 Async-First Design
- **Pattern:** Use `graph.astream_events(..., version="v2")` to provide real-time token streaming to the frontend.
- **Common Mistake:** Blocking the event loop with `time.sleep()` or synchronous database calls inside a node. Use `await asyncio.to_thread()` for legacy DB ops.

### 4b.3 Prompt Engineering Discipline
- **System/User Separation:** Always use `SystemMessage` for instructions and `HumanMessage` for data.
- **Few-Shot:** For the intent classifier, include 3 examples of "FAST" and 3 of "DEEP" in the system prompt to ensure high routing accuracy.
- **Max Tokens:** Always set `max_tokens` explicitly for the Fast Model to prevent "rambling" which increases latency.

### 4b.4 Context Window Management
- **Strategy:** Since Qwen2.5-72B has a large window but MI300X VRAM is partitioned, use a sliding window of the last 10 messages for the Fast Model.
- **Truncation:** Implement a utility to prune older messages if `tokenizer.encode(messages)` exceeds 80% of the partition's `max_position_embeddings`.

### 4b.5 Cost and Latency Budget
- **Latency Target:** Fast Model routing decision < 200ms (TTFT).
- **Caching:** Implement `langchain.cache.InMemoryCache` or Redis-based semantic caching for common queries ("How do I...?") to bypass LLM inference entirely when possible.
- **Routing Ratio:** Aim for >70% of queries to be handled by the Fast Model for common interactions to preserve Big Model throughput.

|-----------|----------|-------------------|-------------|

## 6. Guardrails

### Online Guardrails (Catastrophic)
| Guardrail | Target Failure | Action |
|-----------|----------------|--------|
| **VRAM Partition Guard** | VRAM OOM | Check `nvidia-smi` (or ROCm equivalent) before Big Model call; if >90% partition util, force FAST fallback. |
| **CWE Blocker** | Security Vulnerabilities | Post-process Big Model output with regex for `eval(`, `os.system(`, and raw `execute(f"...")`. |
| **Routing Sanitizer** | Infinite Loops | If `router` returns anything other than FAST/DEEP, default to DEEP and log a routing error. |

### Offline Flywheels (Quality)
| Flywheel | Signal | Action |
|----------|--------|--------|
| **Human-in-the-loop** | "Bad" Routing | Users can "Flag" a response. If flagged "Too slow for simple question", Gemma-4 routing prompt is tuned. |
| **Fallback Audit** | Timeout Frequency | Weekly audit of Big Model timeouts. If >10%, increase hardware partition or optimize RAG retrieval. |
