# Phase 33 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn initialized in `web/components.json` |
| Preset | `b0` (`base-nova`, neutral base, lucide, Inter) from `npx shadcn@latest info` |
| Component library | shadcn/ui + Radix/Base UI primitives; local AI Elements code under `web/src/components/ai-elements/` |
| Icon library | `lucide-react` |
| Font | `Inter, ui-sans-serif, system-ui` via `--font-sans`; mono chips use `JetBrains Mono, monospace` |

Phase 33 must preserve existing UI structure. No new OpenUI, GSAP, runtime attachment upload, shell/session ownership UI, or new registry block adoption. Work is visual pinning plus clearer prompt-context affordances.

Existing relevant surfaces:
- `ChatInput`: keep input shell, mention menu, stop button, drag/drop snippet chip behavior, and prompt placeholder.
- `ChatStream`: keep onboarding entry/reopen summary and user sent artifact rendering.
- `OnboardingGuide`: preserve `IDLE -> QUALIFYING -> STREAMING -> PLAN_READY -> DONE`, cancel, try-again, and completion behavior.
- `PromptInput`: may remain internal, but app-facing submit contract must become typed and must not expose `FormEvent` outside `ChatInput`.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, chip internal icon/text gap, mention token padding |
| sm | 8px | Compact row gaps, chip gap, mention menu item gap |
| md | 16px | Default card/input padding, chat message vertical spacing |
| lg | 24px | Onboarding panel padding, large prompt drop zone spacing |
| xl | 32px | Major onboarding/card gaps |
| 2xl | 48px | Empty-state vertical breathing room |
| 3xl | 64px | Page-level onboarding centering only |

Exceptions: keep `ChatInput` submit button and removable chips at minimum 44px touch target. Existing `min-h-11`, `size-11`, and textarea `min-h-[44px]` are pinned.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`--text-body`) | 400 | 1.5 |
| Label | 14px (`--text-label`) | 400 | 1.4 |
| Heading | 24px (`--text-heading`) | 600 | 1.2 |
| Display | 48px (`--text-display`) | 600 | 1.1 |

Phase-specific type rules:
- Chip text uses Label size. Pre-send chip metadata may use mono only for paths and line ranges.
- Mention menu items use Label size with truncated path text.
- Backend-prompt expansion content is not shown in composer; visible user text remains Body size in sent message.
- Onboarding copy keeps current headings and button labels; tests pin behavior rather than restyling.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` light `#ffffff`, dark `#020605` | Chat workspace background, empty/onboarding backdrop |
| Secondary (30%) | `var(--surface-1)` light `#f0fdfa`, dark `#05110d`; `var(--surface-2)` for nested panels | Cards, onboarding panels, mention menu, prompt input group |
| Accent (10%) | `var(--brand)` light `#059669`, dark `#10b981`; `var(--brand-muted)` for low-emphasis chips | Send button, selected mention row, mention tokens, removable context-chip border/background, focus ring |
| Destructive | `var(--destructive)` light `#e11d48`, dark `#ef4444` | Destructive actions only; none introduced by Phase 33 |

Accent reserved for: send button, selected mention-menu option, `@path` mention pill in sent message, focused input ring, removable context-chip border/background, onboarding primary start/done affordance if already present. Do not apply accent to every hover state.

Semantic color note: chip removal is immediate but non-destructive; use neutral/brand hover opacity, not destructive red. File/folder content load failures appear in backend prompt text and tests, not as a new visible blocking error state.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Send message` for typed submit action; visible icon-only submit keeps accessible label `Send message` |
| Empty state heading | `Ready to start a conversation?` |
| Empty state body | `Start onboarding to generate a repo guide, or ask a question with @path mentions and dropped snippets.` |
| Error state | `Some context could not be loaded. Send continues with failure notes in the prompt context.` |
| Destructive confirmation | None. `Remove context` is immediate with no confirmation or undo. |

Required visible helper copy:
- Composer placeholder stays: `Ask about your code, use @path/to/file, or drop snippet here...`
- Snippet chip label format before send: `Snippet · {path}:{startLine}-{endLine}`.
- File chip label format before send: `File · {path}`.
- Folder chip label format before send: `Folder · {path} · up to 8 files, 8k chars each`.
- Removable chip accessible label: `Remove {type} context: {path}`.
- Chip tooltip/title: `Click to remove {type} context`.
- Mention menu empty state is unchanged if not already shown; Phase 33 must not add new fetch behavior for mentions.

Interaction contract:
- Enter sends only when text/content is non-empty and not loading.
- Shift+Enter inserts newline.
- Mention-menu ArrowUp/ArrowDown, Enter/Tab selection, and Escape close behavior remain unchanged.
- Stop generation button behavior is preserve-only; no new abort/liveness states in this phase.
- Removing file/folder/snippet chips happens immediately and updates sent context preview before send.
- Sent user message shows `displayMessage` plus artifacts; `backendPrompt` remains hidden payload.

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|
| shadcn official | Existing installed primitives only: `button`, `card`, `command`, `input-group`, `textarea`, `tooltip`, `skeleton`, `badge` as already present | not required — existing local code — 2026-05-29 |
| AI Elements local code | Existing `prompt-input`, `message`, `conversation`, `shimmer`, `reasoning`, `tool`, `inline-citation` | not a new registry import — existing checked-in source — 2026-05-29 |
| third-party registries (`@fonttrio`, `@shadcn-studio`, `@ss-*`, `@diceui`) | none for Phase 33 | not used — no `shadcn view` gate required — 2026-05-29 |

Registry rule: If implementation proposes any new third-party block, stop and run `npx shadcn view {block} --registry {registry_url}` before inclusion. Without that gate, the block is out of contract.
