# Phase 26: Admin Dashboard - Research

**Researched:** 2026-05-16
**Domain:** AI Summarization, Admin UI, Role-Based Access Control
**Confidence:** HIGH

## Summary

The Admin Dashboard introduces a repository-specific view of "intern confusion" topics generated from user chat histories and questions. The core summarization engine uses the Fast Model (Gemma-4-9B-it) executed via APScheduler cron jobs. Currently, the backend scheduler in `api/jobs/reports.py` generates global reports. To align with locked decision D-01, the query logic in `api/reports/generator.py` must be refactored to aggregate data on a per-repository basis, generating distinct markdown files (e.g., `daily-{repo_id}-{date}.md`) stored in `ReportsHub`.

For the frontend, the UI will be implemented as a new Next.js App Router page (e.g., `web/src/app/repo/[id]/admin/page.tsx`). It will fetch these pre-computed markdown reports via a new set of API endpoints in `api/routes/admin.py`. Access control (D-03) requires checking an `is_admin` flag. Since no such column currently exists in the `users` table (`sql/migrations/0019_add_orgs_and_users_tables.sql`), an Alembic/SQL migration will be needed to add this boolean flag, coupled with a FastAPI dependency to enforce the authorization.

**Primary recommendation:** Introduce a database migration to add `is_admin` to the `users` table, refactor the APScheduler report generator to map over all active repositories, and build a simple markdown list view in Next.js.

### Anti-Patterns to Avoid
- **Anti-pattern:** Real-time generation of the dashboard. AI calls take time and are expensive; always read the pre-computed files from `ReportsHub` on the dashboard route.
- **Anti-pattern:** Creating a separate `admin_users` table. Instead, add an `is_admin` boolean to the existing `users` table to maintain reference integrity.

## Common Pitfalls

### Pitfall 1: Unprotected Admin Endpoints
**What goes wrong:** Regular users can read administrative insights.
**Why it happens:** Reusing generic routes without adding the `is_admin` dependency.
**How to avoid:** Ensure the new `/admin/repo/{repo_id}/reports` endpoints depend on a `verify_admin` function that checks the database.
**Warning signs:** Endpoints in `admin.py` lacking `Depends(verify_admin)`.

### Pitfall 2: Global Aggregation Leakage
**What goes wrong:** A repository maintainer sees confusion topics from a completely different repository.
**Why it happens:** The SQL query in `api/reports/generator.py` drops the `WHERE repo_id = :repo_id` clause.
**How to avoid:** Explicitly require `repo_id` in the `generate_daily_intelligence_report` function signature.

## Code Examples

### FastAPI Admin Dependency
```python
async def verify_admin(request: Request) -> None:
    user_id = getattr(request.state, "user_id", None)
    if not user_id:
        raise HTTPException(status_code=401, detail="Unauthorized")
        
    engine = get_engine()
    async with engine.connect() as conn:
        result = await conn.execute(
            text("SELECT is_admin FROM users WHERE id = CAST(:uid AS uuid)"), 
            {"uid": user_id}
        )
        row = result.fetchone()
        if not row or not row._mapping["is_admin"]:
            raise HTTPException(status_code=403, detail="Forbidden: Admin access required")
```
